# Chrome DOM Bridge MCP — 構想メモ

自分の普段使いのChrome（プロファイル・ログイン状態そのまま）に対して、
Claude が必要時に DOM 取得・ページ操作・DevTools級情報取得できる軽量MCP。

---

## 1. やりたいこと（ゴール）

- Claude（Code/Desktop）から「今開いてるページ見て」と頼むと、**今のChromeの現タブ**を解析できる
- **自分の普段のChromeをそのまま使う**（プロファイル・ログイン維持）
- セットアップは可能な限りシンプル（初回1回だけ）
- 読み取りだけでなく、**ページ操作**（クリック・入力・遷移）も将来的に可能にする
- メモリ・Network・パフォーマンスなど DevTools 級の情報も取れる余地を残す

### なぜ既存ツールでなく自作か（不満点）

- **Chrome MCP**: 別プロファイルのブラウザが立ち上がる → 普段のログイン状態が消える ✗
- **Playwright MCP**: 毎回ブラウザ起動・キー登録などセットアップが面倒、これも別ブラウザ ✗
- → **既存の自分のChromeに軽く同居して繋ぐだけ**の、もっとシンプルな物が欲しい

---

## 2. 採用アーキテクチャ

```
[Claude] ←①stdio(MCP)→ [バイナリ(ヘルパー)] ←②WebSocket→ [拡張(Chrome内)] → DOM/操作
```

3要素:

- **拡張機能**: 自分の本物Chromeの中で動く。現タブのDOM取得・操作を担当
- **バイナリ(ヘルパー)**: 真ん中の翻訳機。①Claude側はstdioでMCP、②拡張側はWSで通信
- **Claude**: MCPクライアント。バイナリを子プロセスとして起動し、ツールを呼ぶ

### 2本の接続

- **① バイナリ ⇄ Claude（stdio / MCP）**
  - Claude がバイナリを子プロセスで起動（あなたは起動操作しない）
  - stdin/stdout のパイプで MCP 通信。ポート不要
- **② バイナリ ⇄ 拡張（WebSocket）**
  - バイナリが `ws://localhost:9333` 等で待ち受け
  - 拡張(background)が自分から接続しに行く（拡張はポート待ち受け不可なので「繋ぎに行く」側）

### 1リクエストの流れ

```
1. Claude →①stdio→ バイナリ「get_dom 呼ぶ」
2. バイナリ →②WS→ 拡張「今のDOMくれ」
3. 拡張: chrome.scripting で現タブDOM取得
4. 拡張 →②WS→ バイナリ「これDOM」
5. バイナリ →①stdio→ Claude「結果これ」
```

バイナリは stdio ⇄ WS を橋渡しする中継役。

---

## 3. なぜこの構成になったか（検討の結論）

### 拡張は「サーバー」になれない

- 拡張はChromeのサンドボックス内 → ポート待ち受け(listen)が物理的に不可
- WASM でも同じ（ブラウザ内なら権限はJSと同条件。速いだけで強くない）
- → 「待ち受ける誰か」= ブラウザ外のローカルプロセス（バイナリ）が必ず1個要る
- 拡張は外向き接続(fetch/WS connect)はできる → だから拡張からバイナリへ**繋ぎに行く**

### MCPサーバーの「サーバー」= 常駐物ではない（重要な誤解ポイント）

- MCP(stdio型)の「サーバー」= ただの実行ファイル
- **自分で起動しない**。Claude が必要時に自動起動する
- 登録は1回だけ: `claude mcp add dom-bridge /path/to/helper`
- 「サーバー起動という作業が要らない」が正確（起動自体はClaudeが裏でやる）
- 停止も不要（Claude終了でバイナリも終了）

### MCP は必須か？ → 用途次第

- **Claudeが取りに来る窓口が欲しい**（＝今回の要件）→ **MCP必須**。ローカルプロセス1個不可避
- 参考: もし「自作ツールが自分でAIに送る」だけなら MCP不要（バイナリがClaude APIを直接叩く方式もある）が、
  今回は「Claude側から呼ぶ」形なので MCP を採用

### Native Messaging を使わない理由

- Native Messaging = 拡張からローカルバイナリをChromeが起動する公式機能
- 「Chromeがバイナリ起動」だが、MCPは「Claudeがバイナリ起動」 → 起動主が2つになり別プロセスで複雑化
- → **WSブリッジの方が素直**（MCPサーバー＝Claudeが起動する1バイナリ、拡張とはWS）。今回はこちら採用

---

## 4. ユーザー目線の動き

### 初回（1回だけ）

1. 拡張をChromeに入れる（ストア or 手動読込）
2. ヘルパー(バイナリ)を設置
3. Claudeに登録: `claude mcp add dom-bridge /path/to/helper`

### 毎回

1. いつも通りChromeでページを開く（ログイン済みのまま普通に閲覧）
2. Claudeに「今のページのDOM見て」「この要素クリックして」等を頼む
3. Claudeが裏で取得/操作 → 結果を返す

ユーザーがやること = **普通にブラウジングして、Claudeに頼むだけ**。起動・コマンド・キー登録なし。

---

## 5. データの取り方（プル型）

- 常時監視ではない。**頼まれた瞬間だけ取得**（スナップショット＝写真型）
- Claudeが見るのは「呼んだ時点のDOM」。変わったらもう一度呼べば最新
- リアルタイム監視（MutationObserver常時送信）も技術的に可能だが重い・プライバシーリスク大 → 通常不要

---

## 6. 取得・操作できる範囲

### 操作（公開すればClaudeから使える / 足し算で増える）

拡張の能力が天井。そのうち「ハンドラを書いて公開した分」だけClaudeが呼べる。

- `get_dom` … DOM取得
- `click` … セレクタ指定クリック `el.click()`
- `fill` … 入力 `input.value=`（※React等はイベント発火が必要な場合あり）
- `navigate` … URL遷移
- `screenshot` … スクショ

注意: 単発の単純操作は簡単。複数ステップの**確実な**操作は、要素特定の安定性・タイミング待ちで難易度が上がる（Playwrightが解決している領域）。

### 情報取得の3層

- **層1（ページ内API・簡単）**
  - Network: `performance.getEntriesByType('resource')` でURL/サイズ/読込時間
  - メモリ: `performance.memory`（JSヒープ・ざっくり・Chrome限定）
- **層2（chrome.* API・権限要）**
  - Network監視: `chrome.webRequest`（全リクエスト傍受）
  - システムメモリ: `chrome.system.memory` / `chrome.processes`
- **層3（chrome.debugger = CDP・DevTools級）**
  - Network全body・ヘッダ、メモリヒープスナップショット、Performanceトレース等
  - 代償: 対象タブに「デバッグ中」バナー表示、`debugger`権限（強力＝要注意）

用途に応じて層を選ぶ。普段使いは層1〜2で足りることが多い。

## 7. クロスプラットフォーム

- **OS非依存（共通）**: 拡張、ヘルパーのロジック、WS/MCP通信 → 9割共通
- **OS依存（分岐）**:
  - バイナリ形式 → ビルド時に各OS出力（Goクロスコンパイル / `bun --compile` / `pkg`）
  - 設定ファイル置き場 → Mac `~/Library/...` / Win `%APPDATA%`・レジストリ / Linux `~/.config/...`
  - インストーラ → OS毎に用意（やる事は「ファイル置く＋権限」程度）
- 楽にする手: ヘルパーを **Go** で書くと全OSバイナリ生成が1コマンドで配布が平和
- Go採用理由: 単体バイナリ配布しやすい、起動が速い、JSON/HTTP/WS周りが素直、Rustより初心者が詰まりにくい。今回はGo基礎を学びながら小さく作る方針

---

## 8. セキュリティ上の注意（このルートの危険点）

- **DOM/通信を外部AIに送る** = ログイン後の個人情報・社外秘も流れ得る → 送信範囲を絞る、機微サイト除外
- **WSブリッジ** = localhost に他プロセスも繋げる → **トークン認証必須**、localhost限定
- **拡張の権限** = 過大にしない（理想は `activeTab` 中心、必要な分だけ）
- **層3(debugger)** = 強力。常用は避け、必要時のみ
- 個人の手元実験ならアリ。配布・常時稼働は要慎重

---

## 9. 詰まりやすいポイント（実装時）

- WS接続タイミング（拡張↔バイナリの起動前後）→ 拡張側に再接続ループ
- WS待ち受けポート衝突 → 既定は `127.0.0.1:9333` 固定。衝突時はランダム退避せず明確に失敗し、設定ファイル/環境変数と拡張側設定UIでポート変更可能にする
- MV3 の background(service worker)が寝る → WS切れる → keepalive / 再接続
- 拡張ID: 手動配布だと変わる → 設定の整合が要る（ストア公開ならID固定）
- 要素特定の安定性・操作タイミング（自動操作する場合）

## 付録: このセッションの検討経緯（要約ログ）

時系列の議論。結論に至る思考の流れ。

1. **発端**: Chrome MCP以外でDOM見る方法は? 自作できる? wsで何とかできる?
   - → CDP(Chrome DevTools Protocol)をWebSocketで直接叩けば取れる。MCPも内部はこれ。
     `--remote-debugging-port` で起動 → `localhost:9222/json` でWS URL取得 → CDPコマンド送信。

2. **「localhost以外、一般ドメインは?」**
   - → localhostは制御チャンネル。`Page.navigate`で任意ドメインに飛ばせる。localhost専用ではない。
   - 別解: サーバーで無人取得ならPlaywright/Puppeteerヘッドレス。

3. **「作るならどうなる? ローカルにサーバー立ててURL入れる?」**
   - → その通り。ローカルにHTTPサーバー(Express+Playwright)、POSTでURL受けてDOM返す最小構成を提示。

4. **「Chrome拡張にするのは無理?」**
   - → 拡張なら今見てるタブのDOMに content script で直アクセス可。ヘッドレス不要、ログイン状態も使える。

5. **「拡張でローカルサーバー的なものは可能?」**
   - → 不可。拡張はサンドボックス内でポート待ち受け不可。拡張は「クライアント」専門。
     外向きfetch/WS connectはできるが、サーバー役は無理。

6. **「MCP化は無理?」**
   - → 拡張単体では不可（サーバーになれない）。でも「拡張+ローカルMCPサーバー(WSブリッジ)」なら可能。
     拡張が自分からWSに繋ぎに行く形。これが既存のChrome系MCPの正体。

7. **「拡張のボタンでローカルサーバー起動を促せる?」**
   - → Native MessagingかカスタムURLスキームで可能。ただし初回登録1回は不可避（OS権限の外）。
     2回目以降はボタンで起動できる。

8. **「WASMでサーバーにできない?」**
   - → 不可。WASMもブラウザ内なら権限はJSと同条件。壁は「ブラウザがlisten socketを禁止」だから。
     WASI(ブラウザ外ランタイム)ならソケット可だが、結局ローカルプロセス起動と同じ。

9. **「MCP自体サーバー必須? AI連携の他の方法は?」**
   - → MCPは必ず「待ち受ける何か」が要る。が、AI連携が目的なら別手段もある:
     (A)拡張→Claude API直叩き(サーバー不要だがClaude内では使えない)、(B)ファイル経由、(C)MCP。

10. **「API直叩きの意味が不明、Claudeがバイナリ見に来るのでは?」**
    - → 逆。Claude APIはネット上の受け身の窓口。A案は「バイナリが送りつける」方向。
      だがユーザーの要件は「Claudeが取りに来る窓口が欲しい」= MCP型と判明。話が一周。

11. **要件確定**: Chrome MCPはプロファイル消える、Playwright MCPは設定面倒 → もっとシンプルに既存Chromeへ繋ぐMCPが欲しい。
    - → 「拡張+極薄MCPブリッジ」が、自分のプロファイル維持できる唯一筋の良い形。

12. **ユーザー目線の動き / 裏側 / インストーラの役目を確認**
    - インストーラ ≠ サーバー。インストーラはファイル置くだけで終了。
    - MCPサーバーは**Claudeが起動**（`claude mcp add`で登録）。Native Messaging混同を訂正。
    - 停止不要（Claude終了で落ちる）。

13. **「MCPサーバーはどこに? サーバー起動じゃなくていい?」**
    - → MCP(stdio型)の「サーバー」= ただの実行ファイル。自分で起動せず、Claudeが自動起動。
      「起動作業が要らない」が正確。

14. **「リアルタイムで見えてる?」** → No。プル型(オンデマンド)。頼んだ瞬間のスナップショット。

15. **「Chrome APIは使える? ボタン押せる?」** → 使える。読取専用でない。クリック・入力・遷移・タブ操作可。
    公開したツールの分だけClaudeが使える。

16. **「操作も簡単?」** → 単純操作は簡単。複数ステップの確実な操作は要素特定・待機で難易度上がる。

17. **「クロスプラットフォームは?」** → 拡張・ロジックは共通。バイナリ形式・設定パス・インストーラだけOS分岐。Go推奨。

18. **「バイナリの仕組み? バイナリ⇄ws⇄Chrome?」** → 正確には2本: ①stdio(Claude) ②WS(拡張)。バイナリは中継翻訳機。

19. **「拡張でできること全部できる?」** → 原理的にYes。ただし公開したツールの分だけ(足し算)。

20. **「メモリ・Networkも分かる?」** → 取れる。深さで3層(ページ内API / chrome.* / chrome.debugger=CDP)。
